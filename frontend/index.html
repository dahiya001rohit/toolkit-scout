    The agent behind this

    How each app gets researched

    1. Find the docs — start from the given hint, try the conventional homes (docs.X, developers.X, api.X), mine api/developer links out of every page that loads, and only then let the LLM suggest URLs. A wrong guess just 404s — the LLM can point at pages, it can never be the evidence.
    2. Fetch the real pages — async httpx pulls up to three documentation pages per app.
    3. Extract, grounded — the LLM must fill a strict pydantic schema using only the fetched text. Not in the text → unknown. Every field carries the URL that backs it.
    4. Checkpoint everything — each finished app is written to disk immediately. Crashes and quota deaths resume; nothing restarts.
    5. Rotate models on quota — Groq's free limits are per-model, so a daily-quota 429 swaps models mid-run instead of killing the batch.

    Then a second, skeptical pass

    1. Re-fetch every app's evidence pages and re-extract with an auditor prompt on a different model.
    2. Diff the five error-prone fields against pass 1.
    3. A tie-break call settles each disagreement and must quote the deciding sentence from the docs. Unresolvable → pass 1 stands, labelled as such.

    Where a human was needed

    • The quota call — when Groq's daily token budget died at app 27 of 100, choosing between paying, waiting a day, or building the model pool was a human decision; the agent then executed it.
    • A corrupted checkpoint — an IDE auto-format split the checkpoint file mid-run. A human spotted the crash cause; the agent repaired the file by stream-parsing it back together.
    • Prompt quality — pass 1 returned lazy descriptions ("a productivity app"). A human caught it; one added prompt rule fixed it.
    • The hand-audit — a random sample was checked against the live docs by a person (results in section 05).
    • Judgment on traps — deciding that "no public API, and here's the evidence" is a correct finding, not a failure to route around.

    Known misses & limits (kept honestly)

    • PitchBook — no documentation page could be fetched at all (site blocks bots; API is partnership-gated). Reported as an evidenced failure, not guessed.
    • Pass 1 ran on two models (quota rotation): 27 apps on llama-3.3-70b, the rest on gpt-oss-120b — disclosed because it makes rows slightly inconsistent, and exploited by verifying cross-model.
    • Fetched text is truncated (~4k chars/page): breadth judgments lean on docs landing pages more than full endpoint references.
